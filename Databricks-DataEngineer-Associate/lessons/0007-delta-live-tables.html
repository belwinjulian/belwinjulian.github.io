Delta Live Tables (DLT)

1. What is DLT and why does it matter?

Building a multi-stage pipeline normally means writing and managing multiple notebooks yourself: run Bronze first, then Silver, then Gold, retry on failure, track schema changes, validate data quality. That is a lot of plumbing code.

Delta Live Tables (DLT) is Databricks' declarative pipeline framework. You write SQL or Python that says what each table should contain — DLT figures out the execution order, manages checkpoints, handles retries, and enforces data quality rules automatically.

Without DLT With DLT ───────────────────────────────────────────────────── You manage: DLT manages: • Notebook execution order • Dependency resolution • Retries on failure • Automatic retries • Checkpoint files • Checkpointing • Schema change handling • Schema evolution • Data quality checks (custom) • Built-in Expectations • Cluster lifecycle • Cluster lifecycle You write: job orchestration You write: table definitions

A pipeline is a group of DLT notebooks or SQL files. When you run a pipeline, DLT builds a dependency graph of all your table definitions and executes them in the correct order.

DLT is declarative — you define tables, not steps. This is the key distinction from regular Spark notebooks where you write imperative code to run transformations manually.

2. Two table types: Streaming Tables vs. Materialized Views

Every DLT table is one of two types. Choosing the right one is a core exam question.

Streaming Table

Processes new data incrementally. Only rows that arrived since the last pipeline run are processed — previously seen rows are not reprocessed. This is the correct type when data flows in continuously over time (Auto Loader files, Kafka events, CDC feeds).

Materialized View

Recomputed from all source data on each pipeline update. Think of it as a stored view — the query result is persisted to a Delta table. Use when the output must reflect the full picture (aggregations, joins across the whole dataset).

Streaming Table Materialized View ───────────────────────────────────────────────────────────── Source Append-only stream Any table or query Processing Incremental (new rows) Full recompute each run State Checkpoint tracks offset Query re-runs fully Best for Bronze ingestion, Gold aggregations, row-level Silver filter cross-table joins, metrics dashboards SQL keyword STREAMING TABLE MATERIALIZED VIEW
Decision rule: Is new data always appended? → Streaming Table.
Does the result depend on all historical rows? → Materialized View.
A Streaming Table can read from another Streaming Table using STREAM(LIVE.table). A Materialized View reads from a Materialized View or a Streaming Table using LIVE.table (no STREAM wrapper). Mixing these up is a common exam trap.

3. DLT SQL syntax

The exam is SQL-first. Learn these two statements cold.

Creating a Streaming Table (Bronze — raw ingestion)

CREATE OR REFRESH STREAMING TABLE bronze_orders
AS SELECT * FROM STREAM read_files(
  's3://my-bucket/raw/orders/',
  format => 'json'
);

Breaking that down:

  • CREATE OR REFRESH STREAMING TABLE — creates the table if it doesn't exist, otherwise refreshes it incrementally
  • STREAM read_files(...) — reads files from cloud storage as a stream (DLT's SQL-native equivalent of Auto Loader)
  • format => 'json' — the file format (json, csv, parquet, avro…)

The LIVE. prefix — referencing DLT tables

Inside a DLT pipeline, you reference other DLT tables with the LIVE. prefix. This tells DLT "this is a table managed by this pipeline" and is how the dependency graph is built.

-- Silver: reads from the Bronze streaming table, filters bad rows
CREATE OR REFRESH STREAMING TABLE silver_orders
AS SELECT *
FROM STREAM(LIVE.bronze_orders)   -- STREAM() wrapper = read as stream
WHERE amount > 0
  AND customer_id IS NOT NULL;
LIVE.table_name = reference to another DLT table in the same pipeline.
STREAM(LIVE.table_name) = read that DLT table as a stream (incrementally).
Forget LIVE. and the table is treated as an external table — DLT won't manage the dependency.

Creating a Materialized View (Gold — aggregation)

-- Gold: aggregated revenue per day (reads ALL rows from Silver each run)
CREATE OR REFRESH MATERIALIZED VIEW gold_daily_revenue
AS SELECT
  date(order_ts)    AS order_date,
  SUM(amount)       AS total_revenue,
  COUNT(*)          AS order_count
FROM LIVE.silver_orders          -- no STREAM() wrapper = full recompute
GROUP BY date(order_ts);

Full Medallion pipeline in SQL

read_files() STREAM(LIVE.bronze) LIVE.silver │ │ │ ▼ ▼ ▼ ┌────────┐ ┌──────────┐ ┌──────────┐ │ Bronze │─────────────▶ Silver │────────────▶ Gold │ │Streaming│ │Streaming │ │Mat. View │ └────────┘ └──────────┘ └──────────┘ raw JSON from filter + clean aggregate revenue cloud storage bad rows dropped per day

4. DLT Python syntax

Python DLT uses decorators. Each function becomes a DLT table — the function name is the table name.

import dlt
from pyspark.sql.functions import col

# Bronze: ingest from Auto Loader (streaming)
@dlt.table(comment="Raw orders from cloud storage")
def bronze_orders():
    return (spark.readStream
            .format("cloudFiles")
            .option("cloudFiles.format", "json")
            .option("cloudFiles.schemaLocation", "/checkpoints/orders_schema")
            .load("s3://my-bucket/raw/orders/"))

# Silver: read Bronze as a stream, filter bad rows
@dlt.table
def silver_orders():
    return (dlt.read_stream("bronze_orders")   # incremental read
            .filter("amount > 0")
            .filter(col("customer_id").isNotNull()))

# Gold: aggregate Silver (full recompute)
@dlt.table
def gold_daily_revenue():
    return (dlt.read("silver_orders")          # full read (no _stream)
            .groupBy("order_date")
            .agg({"amount": "sum"}))
dlt.read_stream("table") = incremental read of a Streaming Table (like STREAM(LIVE.table) in SQL).
dlt.read("table") = full read for a Materialized View (like LIVE.table in SQL).
The function name becomes the table name. No quotes, no LIVE. needed — that's the Python equivalent.

5. Data quality with Expectations

Expectations are DLT's built-in data quality constraints. You define a rule and specify what happens when a row violates it. This replaces writing your own quality checks in Spark code.

There are three behaviors on violation:

Behavior What happens to violating rows Pipeline ──────────────────────────────────────────────────────────── WARN (default) Row is kept, violation is recorded Continues DROP ROW Row is silently dropped Continues FAIL UPDATE Row is rejected, pipeline stops Stops

SQL Expectations syntax

CREATE OR REFRESH STREAMING TABLE silver_orders (
  CONSTRAINT valid_amount
    EXPECT (amount > 0),                          -- warn (default)

  CONSTRAINT has_customer
    EXPECT (customer_id IS NOT NULL)
    ON VIOLATION DROP ROW,                        -- drop bad rows

  CONSTRAINT recent_order
    EXPECT (order_date >= '2020-01-01')
    ON VIOLATION FAIL UPDATE                      -- stop the pipeline
)
AS SELECT * FROM STREAM(LIVE.bronze_orders);

Python Expectations syntax

@dlt.expect("valid_amount", "amount > 0")                     # warn
@dlt.expect_or_drop("has_customer", "customer_id IS NOT NULL") # drop
@dlt.expect_or_fail("recent_order", "order_date >= '2020-01-01'") # fail
@dlt.table
def silver_orders():
    return dlt.read_stream("bronze_orders")
The three Python decorators map directly to the three behaviors:
@dlt.expect → warn  |  @dlt.expect_or_drop → drop  |  @dlt.expect_or_fail → fail
If the exam says "rows that violate quality rules should not appear in the output but the pipeline should keep running" — that is DROP ROW.
Expectation violations are tracked in the DLT event log and visible in the pipeline UI as metrics. WARN violations do NOT stop the pipeline — they are surfaced for monitoring only.

6. Pipeline modes and updates

Development vs. Production mode

When you create a DLT pipeline you choose a mode:

  • Development mode — cluster is reused between runs (faster iteration), pipeline does NOT auto-restart on failure. Use while building and testing.
  • Production mode — fresh cluster each run (isolated), pipeline auto-restarts on recoverable failures. Use when scheduling the pipeline to run on a schedule.
Development mode reuses the cluster to save startup time. Production mode uses a fresh cluster for isolation and has auto-retry. The exam will ask "which mode auto-restarts on failure?" — Production.

Update types

When you trigger a pipeline update (a run), you choose how it processes data:

  • Triggered update — process all new data once, then stop. The pipeline runs, catches up, and shuts down.
  • Continuous update — pipeline stays running and processes data as it arrives with low latency.
  • Full refresh — clears all existing table data and reprocesses everything from scratch. Use when you change the pipeline logic and need to recompute history.
Full refresh is the answer when the exam says "the pipeline logic changed and the existing table data is incorrect." It wipes and recomputes. For normal incremental operation, use Triggered (batch-style) or Continuous (streaming).

DLT pipeline vs. Lakeflow Jobs

These solve different problems:

DLT Pipeline Lakeflow Job ──────────────────────────────────────────────────────────── What it manages Table dependencies Task dependencies What you define Table SQL/Python Notebook/task order Data quality Built-in Expectations Custom code Best for Multi-table pipelines Orchestrating mixed tasks with quality rules (notebooks, SQL, Python)

A common pattern is to trigger a DLT pipeline from a Lakeflow Job as one task in a larger workflow.
