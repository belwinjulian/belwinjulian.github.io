Delta Lake Fundamentals

1. Quick recap: what Delta Lake stores on disk

From Lesson 1: Delta Lake stores two things side by side in cloud storage:

my_table/ ├── part-00000.parquet ← actual data (Parquet files) ├── part-00001.parquet └── _delta_log/ ├── 00000000000000000000.json ← transaction 0 (table created) ├── 00000000000000000001.json ← transaction 1 (first INSERT) └── 00000000000000000002.json ← transaction 2 (UPDATE ran)

Every write — INSERT, UPDATE, DELETE — appends a new JSON entry to _delta_log/. Delta never overwrites old Parquet files immediately; it writes new ones and records the change in the log. This is what makes ACID possible, and it's also what enables Time Travel.

2. Time Travel

Because Delta Lake keeps every version of your data in the transaction log, you can query any past version of a table. This is called Time Travel.

There are two ways to specify which version you want:

By version number

-- See the table as it was at version 3
SELECT * FROM orders VERSION AS OF 3;

By timestamp

-- See the table as it was on a specific date
SELECT * FROM orders TIMESTAMP AS OF '2026-05-01';

Check what versions exist

-- Lists every transaction: who ran it, when, what operation
DESCRIBE HISTORY orders;

DESCRIBE HISTORY is your go-to command when you need to audit a table or find a version to restore to. It shows you the version number, timestamp, operation type (WRITE, UPDATE, DELETE, MERGE), and who made the change.

The exam loves DESCRIBE HISTORY. If a question asks "how do you see all past changes to a Delta table?" — that's your answer.

Restoring a table to a past version

Time Travel is read-only by default — you're just querying the past. To actually roll back the table, use RESTORE:

-- Roll the table back to version 2
RESTORE TABLE orders TO VERSION AS OF 2;

-- Or roll back to a timestamp
RESTORE TABLE orders TO TIMESTAMP AS OF '2026-05-01';
SELECT ... VERSION AS OF = read the past (table is unchanged).
RESTORE TABLE ... TO VERSION AS OF = actually roll the table back.

3. OPTIMIZE and VACUUM

Over time, lots of small writes create lots of small Parquet files. Small files are slow to query. Two commands fix this:

OPTIMIZE — compact small files

OPTIMIZE orders;

OPTIMIZE combines many small Parquet files into fewer, larger ones. This makes queries faster. It does not delete any data — it just reorganises the files.

OPTIMIZE can also use Z-ordering to co-locate related data on disk:
OPTIMIZE orders ZORDER BY (customer_id);
This speeds up queries that filter on customer_id. The exam may ask what Z-ordering does: it organises data on disk so rows with the same column values are stored near each other.

VACUUM — remove old files

VACUUM orders;

VACUUM permanently deletes old Parquet files that are no longer referenced by the current version of the table. By default it keeps files from the last 7 days (168 hours) so that Time Travel still works within that window.

-- You can lower the retention, but this breaks Time Travel for deleted versions
VACUUM orders RETAIN 24 HOURS;
Three things to know cold about VACUUM:
1. It permanently deletes files — irreversible.
2. Default retention = 7 days.
3. Running VACUUM cuts off Time Travel for versions older than the retention window.

4. Schema enforcement vs. schema evolution

Delta Lake is strict about the shape (schema) of data by default.

Schema enforcement (the default)

If you try to write data with a different schema than the table — a new column, a wrong data type — Delta will reject the write and throw an error. The table is protected. This is schema enforcement.

-- This INSERT will FAIL if 'discount' column doesn't exist in orders
INSERT INTO orders SELECT *, 0.1 AS discount FROM new_data;

Schema evolution (opt-in)

Sometimes you legitimately want to add a new column. You can tell Delta to accept it and update the schema automatically using mergeSchema:

-- Python: allow new columns to be added
df.write.option("mergeSchema", "true").mode("append").saveAsTable("orders")
-- SQL: set the spark config to allow schema evolution for this session
SET spark.databricks.delta.schema.autoMerge.enabled = true;
Schema enforcement = Delta rejects mismatched writes (default, safe).
Schema evolution = Delta accepts new columns and updates the schema (opt-in, using mergeSchema).

5. Managed vs. External tables

When you create a Delta table in Databricks, you have two choices for where the data files live:

Managed table (default)

CREATE TABLE orders (
  order_id INT,
  customer STRING,
  amount DOUBLE
);

Databricks controls the file location (inside a default storage path it manages). When you DROP TABLE orders, the data files are deleted permanently.

External table

CREATE TABLE orders_external
LOCATION 's3://my-bucket/orders/';

You specify the file location. When you DROP TABLE orders_external, the table definition is removed but the data files in S3/ADLS remain untouched.

The key exam distinction:
Managed: DROP TABLE = data is deleted.
External: DROP TABLE = only the metadata is removed; files survive.

If a question says "the team accidentally dropped a table and lost data" — they had a managed table.
MANAGED TABLE EXTERNAL TABLE ┌───────────────────┐ ┌───────────────────┐ │ Databricks owns │ │ You own the path │ │ the file location │ │ (S3, ADLS, GCS) │ └────────┬──────────┘ └────────┬──────────┘ │ DROP TABLE │ DROP TABLE ▼ ▼ FILES DELETED FILES SURVIVE (gone forever) (metadata removed only)
