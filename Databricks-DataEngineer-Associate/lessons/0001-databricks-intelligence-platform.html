The Databricks Intelligence Platform & the Lakehouse

1. The problem: why did Databricks exist at all?

Before Databricks, companies had two imperfect choices for storing data:

  • Data Warehouse (e.g. Snowflake, Redshift) — fast SQL queries, reliable, but expensive and rigid. It only handled structured data (neat tables). Raw files, images, and semi-structured JSON had no place here.
  • Data Lake (e.g. raw files in Amazon S3 or Azure ADLS) — cheap, flexible, stores anything. But no ACID guarantees — you could corrupt data and not know it. Queries were slow and unreliable.

Neither was enough. The warehouse was too rigid; the lake was too unreliable.

A Data Warehouse gives you reliability but locks you into structured data.
A Data Lake gives you flexibility but no guarantees. The Lakehouse gives you both.

2. The Lakehouse: one architecture to rule them all

A Lakehouse is an architecture that sits on top of cheap cloud storage (S3, ADLS, GCS) and adds the reliability of a data warehouse on top. You keep your raw files exactly where they are — no copying to a separate warehouse — but now you get ACID transactions, schema enforcement, and fast queries.

The secret ingredient that makes this possible is Delta Lake.

┌─────────────────────────────────────────────────────┐ │ DATABRICKS PLATFORM │ │ Notebooks · Workflows · SQL · ML · Streaming │ ├─────────────────────────────────────────────────────┤ │ DELTA LAKE │ │ (ACID transactions · Schema enforcement · │ │ Time travel · Transaction log) │ ├─────────────────────────────────────────────────────┤ │ CLOUD OBJECT STORAGE │ │ S3 · Azure Data Lake · GCS │ └─────────────────────────────────────────────────────┘ ↑ This whole stack = the Lakehouse

Delta Lake is not a separate database. It is a storage layer — a set of rules and a transaction log written alongside your Parquet files on cloud storage. The files stay where they are; Delta Lake just makes them trustworthy.

Delta Lake stores data as Parquet files plus a _delta_log/ transaction log folder. The log is what gives you ACID properties. If the exam asks "what format does Delta Lake use?" — Parquet files + JSON transaction log.

3. What ACID actually means

ACID sounds intimidating. Here is what each letter means in plain language:

  • Atomicity — a write either fully succeeds or fully fails. No partial updates.
  • Consistency — data always matches the schema rules. Bad data is rejected.
  • Isolation — two people writing at the same time don't corrupt each other's work.
  • Durability — once a write is committed, it stays committed (even if a server crashes).

In practice: you can trust Delta tables. If a job fails halfway through writing, your table is not corrupted.

4. The Medallion Architecture: Bronze, Silver, Gold

Most Databricks projects organise data into three layers. This is called the Medallion Architecture. Think of it as a refinery that progressively cleans raw ore into pure metal.

RAW DATA IN CLEAN DATA OUT (files, APIs, DBs) (business-ready) ┌──────────┐ transform ┌──────────┐ transform ┌────────┐ │ BRONZE │ ──────────▶ │ SILVER │ ──────────▶ │ GOLD │ └──────────┘ └──────────┘ └────────┘ Raw ingestion Cleaned & Aggregated / No changes validated reporting-ready "as-is" copy Deduplicated Business metrics
  • Bronze — Raw data, exactly as it arrived. CSV, JSON, database dumps. Keep everything; delete nothing. This is your audit trail.
  • Silver — Cleaned, deduplicated, validated. A row that was null in Bronze is filtered out here. Schema is enforced.
  • Gold — Aggregated, business-ready. Dashboards, reports, and ML models read from Gold. It might be daily sales totals, user retention metrics, etc.
The exam will present scenarios asking which layer data belongs in. The rule: if it's untouched raw input → Bronze. If it's cleaned but still granular → Silver. If it's aggregated for business use → Gold.

5. The Databricks platform components

Databricks is the company that created Delta Lake and built a cloud platform on top of the Lakehouse pattern. Here are the core components you will see on the exam:

  • Workspace — the web UI where you write code, create notebooks, and manage jobs. Think of it as your project folder in the cloud.
  • Cluster — a group of virtual machines that actually run your Spark code. You spin clusters up to compute, then shut them down to save money. Two types: All-Purpose (interactive, you use while working) and Job (automated, for production runs).
  • Notebook — a document with runnable code cells (Python, SQL, or Scala). This is where you write and test transformations.
  • Apache Spark — the distributed computation engine under the hood. When you write SELECT or a DataFrame operation, Spark executes it across the cluster. You do not need to manage Spark directly — Databricks manages it.
  • Unity Catalog — the governance layer. Controls who can read/write which tables and tracks data lineage.
  • Delta Lake — as above: the reliable storage layer on top of cloud files.
The exam is SQL-first. When a question says "write a transformation," expect SQL. Python (PySpark) appears when SQL can't express the operation.
