DataFrame Operations: Joins, Dedup & Aggregations

1. Joins

A join combines two DataFrames (or tables) by matching rows on a shared key column. This is one of the most tested topics in Domain 3.

The four join types you need to know

orders table customers table ┌────┬──────────┐ ┌──────────┬───────┐ │ id │ cust_id │ │ cust_id │ name │ ├────┼──────────┤ ├──────────┼───────┤ │ 1 │ 101 │ │ 101 │ Alice │ │ 2 │ 102 │ │ 102 │ Bob │ │ 3 │ 999 │ │ 103 │ Carol │ └────┴──────────┘ └──────────┴───────┘ order 3 has no match (999 not in customers) Carol has no orders (103 not in orders)
  • INNER join — only rows that match in both tables. Order 3 and Carol are both dropped.
  • LEFT join — all rows from the left table, matched rows from the right. Order 3 is kept (Carol's fields are NULL). Carol is dropped.
  • RIGHT join — all rows from the right table, matched rows from the left. Carol is kept (order fields are NULL). Order 3 is dropped.
  • FULL OUTER join — all rows from both tables. Order 3 and Carol are both kept, with NULLs where there's no match.

SQL syntax

-- Inner join (default when you just write JOIN)
SELECT o.id, o.amount, c.name
FROM orders o
JOIN customers c ON o.cust_id = c.cust_id;

-- Left join
SELECT o.id, o.amount, c.name
FROM orders o
LEFT JOIN customers c ON o.cust_id = c.cust_id;

-- Full outer join
SELECT o.id, o.amount, c.name
FROM orders o
FULL OUTER JOIN customers c ON o.cust_id = c.cust_id;

PySpark syntax

orders.join(customers, on="cust_id", how="inner")   # or how="left", "right", "outer"

# When the join key has the same name in both DataFrames, pass it as a string.
# When the names differ, pass a condition:
orders.join(customers,
            orders.cust_id == customers.customer_id,
            how="left")
The default join type in both SQL (JOIN) and PySpark (how="inner") is INNER. If the exam shows a JOIN with no qualifier, assume inner.
A LEFT join is the most common exam scenario: "keep all rows from table A even if there's no match in table B." The unmatched rows from B come back as NULL.

2. Deduplication

Duplicate rows are a real-world problem — the same event can arrive twice from a source system. Spark gives you two tools to remove them.

dropDuplicates — remove exact duplicate rows

# SQL
SELECT DISTINCT * FROM orders;

# PySpark — remove rows where every column is identical
df.distinct()

# PySpark — remove rows where specific columns are identical
# (keeps the first occurrence)
df.dropDuplicates(["order_id"])

dropDuplicates(["order_id"]) keeps one row per unique order_id, even if other columns differ. It is the PySpark equivalent of deduplicating on a key column, not the whole row.

distinct() = whole-row dedup (all columns must match).
dropDuplicates(["col"]) = key-column dedup (dedup on specific columns only).

Window functions for dedup (rank-based)

Sometimes you want to keep the most recent duplicate rather than just any one. The pattern is: rank rows within a group, then keep only rank 1.

from pyspark.sql.functions import row_number
from pyspark.sql.window import Window

# For each order_id, rank rows by updated_at descending (newest first)
window = Window.partitionBy("order_id").orderBy(col("updated_at").desc())

df.withColumn("rn", row_number().over(window)) \
  .filter("rn = 1") \
  .drop("rn")

Breaking that down:

  • partitionBy("order_id") — restart the ranking for each unique order_id
  • orderBy(col("updated_at").desc()) — newest row gets rank 1
  • row_number().over(window) — assign the rank
  • .filter("rn = 1") — keep only the newest row per order_id
  • .drop("rn") — remove the helper column
When the exam scenario says "keep the most recent record per key" — think window function + row_number + filter rn = 1. distinct() alone can't do this because it doesn't know which duplicate to keep.

3. Advanced aggregations

You saw basic groupBy().agg() in Lesson 5. Here are the patterns that appear most often on the exam.

Multiple aggregations in one pass

# SQL
SELECT customer,
       COUNT(*)        AS order_count,
       SUM(amount)     AS total,
       AVG(amount)     AS avg_order,
       MAX(amount)     AS largest_order
FROM orders
GROUP BY customer;

# PySpark
from pyspark.sql.functions import count, sum, avg, max

df.groupBy("customer").agg(
    count("*").alias("order_count"),
    sum("amount").alias("total"),
    avg("amount").alias("avg_order"),
    max("amount").alias("largest_order")
)

HAVING — filter after grouping

# SQL — HAVING filters the result of GROUP BY (WHERE filters before grouping)
SELECT customer, COUNT(*) AS order_count
FROM orders
GROUP BY customer
HAVING COUNT(*) > 5;

# PySpark — use .filter() after .agg()
df.groupBy("customer") \
  .agg(count("*").alias("order_count")) \
  .filter("order_count > 5")
WHERE filters rows before grouping.
HAVING filters groups after aggregation.
In PySpark: .filter() before .groupBy() = WHERE. .filter() after .agg() = HAVING.

countDistinct — count unique values

# SQL
SELECT customer, COUNT(DISTINCT product_id) AS unique_products
FROM orders
GROUP BY customer;

# PySpark
from pyspark.sql.functions import countDistinct

df.groupBy("customer") \
  .agg(countDistinct("product_id").alias("unique_products"))

4. Window functions

Window functions compute a value for each row based on a group of related rows — without collapsing the rows the way groupBy does. You get one output row per input row.

groupBy + agg Window function ────────────────────── ────────────────────────────── 3 customers → 3 rows 3 customers × N orders → N rows (rows collapsed) (rows kept, rank added as column)

Common window functions

  • row_number() — unique sequential rank (no ties: 1, 2, 3, 4…)
  • rank() — rank with gaps for ties (1, 2, 2, 4…)
  • dense_rank() — rank without gaps for ties (1, 2, 2, 3…)
  • lag("col", n) — value from n rows before in the window
  • lead("col", n) — value from n rows ahead in the window

SQL window syntax

SELECT customer, amount,
       ROW_NUMBER() OVER (PARTITION BY customer ORDER BY amount DESC) AS rn
FROM orders;

PySpark window syntax

from pyspark.sql.functions import row_number, rank, lag
from pyspark.sql.window import Window

w = Window.partitionBy("customer").orderBy(col("amount").desc())

df.withColumn("rn",   row_number().over(w)) \
  .withColumn("rnk",  rank().over(w)) \
  .withColumn("prev", lag("amount", 1).over(w))
PARTITION BY in a window = restart the calculation for each group (like GROUP BY, but rows are not collapsed).
ORDER BY in a window = defines the order within each partition for ranking or lag/lead.

5. MERGE INTO — upsert pattern

The most powerful Delta write operation: MERGE INTO (also called an upsert — update if exists, insert if not). This is heavily tested because it is the standard pattern for keeping Silver tables in sync with incoming Bronze data.

MERGE INTO silver_customers AS target
USING bronze_updates AS source
ON target.customer_id = source.customer_id

WHEN MATCHED THEN
  UPDATE SET
    target.name   = source.name,
    target.email  = source.email

WHEN NOT MATCHED THEN
  INSERT (customer_id, name, email)
  VALUES (source.customer_id, source.name, source.email);

Breaking that down:

  • MERGE INTO target USING source ON key — match rows between two tables on a key
  • WHEN MATCHED — row exists in both tables → UPDATE
  • WHEN NOT MATCHED — row is in source but not in target → INSERT
  • You can also add WHEN NOT MATCHED BY SOURCE THEN DELETE to delete rows that disappeared from the source
MERGE INTO is the exam's go-to answer for "how do you update existing rows and insert new ones in a single operation?" The keyword is upsert. If you see that word — MERGE INTO.
