PySpark & Spark SQL Basics

1. Two languages, one engine

Spark lets you write data transformations in two styles — and they produce identical results because they run on the same engine underneath:

  • Spark SQL — plain SQL statements. The exam is SQL-first, so always learn the SQL way first.
  • PySpark (Python API) — a Python library where you chain method calls on a DataFrame object. Used when SQL can't express the operation cleanly.
Your code ┌──────────────────┐ ┌──────────────────┐ │ Spark SQL │ │ PySpark │ │ SELECT name │ │ df.select("name")│ │ FROM orders │ │ .filter(...) │ │ WHERE amount>50 │ │ .show() │ └────────┬─────────┘ └────────┬─────────┘ │ │ └──────────┬────────────┘ ▼ Apache Spark engine (same execution plan)
The exam mixes SQL and PySpark questions freely. If you can read both styles, you can answer both types. The logic is the same — only the syntax changes.

2. The DataFrame — Spark's core data structure

A DataFrame is Spark's version of a table: rows and columns, with a defined schema. Every transformation you do in PySpark returns a new DataFrame — you never modify the original.

There are two ways to get a DataFrame:

Load a Delta table as a DataFrame

df = spark.table("orders")          # read a registered table by name
df = spark.read.table("orders")     # same thing, alternative syntax

Run SQL and get the result as a DataFrame

df = spark.sql("SELECT * FROM orders WHERE amount > 50")

spark.sql() is the bridge between SQL and PySpark — you write SQL inside the string, and you get a DataFrame back that you can keep chaining methods on.

spark.table("name") and spark.sql("SELECT * FROM name") both give you a DataFrame. Know both — the exam uses both forms.

3. Core transformations

A transformation is any operation that produces a new DataFrame from an existing one. Nothing actually runs yet — Spark just builds up a plan. (More on this in section 6.)

select — choose columns

# SQL
SELECT order_id, customer, amount FROM orders;

# PySpark
df.select("order_id", "customer", "amount")

filter / where — keep rows that match a condition

# SQL
SELECT * FROM orders WHERE amount > 50;

# PySpark — filter and where are identical, use whichever reads clearer
df.filter("amount > 50")
df.where("amount > 50")
df.filter(df.amount > 50)          # column object syntax

withColumn — add or replace a column

# SQL
SELECT *, amount * 1.1 AS amount_with_tax FROM orders;

# PySpark
from pyspark.sql.functions import col

df.withColumn("amount_with_tax", col("amount") * 1.1)

withColumn either adds a new column (if the name is new) or replaces an existing column (if the name already exists). This is a common exam question.

drop — remove a column

# SQL — no direct equivalent; use SELECT with columns listed explicitly

# PySpark
df.drop("internal_id", "raw_json")

withColumnRenamed — rename a column

# SQL
SELECT order_id AS id, customer AS name FROM orders;

# PySpark
df.withColumnRenamed("order_id", "id") \
  .withColumnRenamed("customer", "name")
withColumn("x", expr) with an existing column name replaces that column — it does not add a second column named "x". This trips people up on the exam.

4. Aggregations

Aggregations reduce many rows down to summary values. In PySpark you always pair groupBy with agg (or a shorthand method).

SQL

SELECT customer, COUNT(*) AS order_count, SUM(amount) AS total
FROM orders
GROUP BY customer;

PySpark

from pyspark.sql.functions import count, sum

df.groupBy("customer") \
  .agg(
    count("*").alias("order_count"),
    sum("amount").alias("total")
  )

Common aggregation functions you need to know:

  • count("*") — count all rows
  • sum("col") — total
  • avg("col") — average
  • min("col") / max("col") — min and max
  • countDistinct("col") — count of unique values
In PySpark, .alias("name") is the equivalent of SQL's AS name. You'll see it inside .agg() calls.

5. Sorting

# SQL
SELECT * FROM orders ORDER BY amount DESC;

# PySpark — orderBy and sort are identical
df.orderBy("amount", ascending=False)
df.sort("amount", ascending=False)

# Multiple columns
df.orderBy(col("amount").desc(), col("customer").asc())

6. Lazy evaluation — the most important concept in Spark

This is the concept that trips up almost every Spark beginner and shows up on the exam.

When you write a PySpark transformation like df.filter(...).select(...).withColumn(...), Spark does not run any of it yet. It just builds up a plan — a recipe for what to do. Nothing touches the data.

Spark only actually executes when you call an action. An action is anything that needs to return a result to you or write data somewhere.

TRANSFORMATIONS (lazy — just building the plan) ────────────────────────────────────────────── df.filter(...) no data movement df.select(...) no data movement df.groupBy(...).agg() no data movement df.join(...) no data movement ACTIONS (trigger execution — Spark actually runs) ────────────────────────────────────────────────── df.show() prints rows to screen df.count() returns a number df.collect() returns all rows to driver (careful: memory!) df.write.saveAsTable() writes to a Delta table
If the exam asks "when does Spark actually execute your code?" — the answer is: when an action is called. Transformations are lazy; actions trigger execution.

Why lazy evaluation exists

Spark uses the built-up plan to optimise before running. If you filter then select, Spark can reorder those steps to push the filter as early as possible and avoid reading unnecessary data. You get better performance without thinking about it.

7. Chaining — the PySpark style

PySpark is designed to be chained. Each method returns a new DataFrame, so you can stack operations cleanly:

result = (
    spark.table("orders")
    .filter("amount > 50")
    .withColumn("amount_with_tax", col("amount") * 1.1)
    .select("order_id", "customer", "amount_with_tax")
    .orderBy("amount_with_tax", ascending=False)
)

result.show(10)          # action — executes everything above

The parentheses around the whole chain let you split it across multiple lines for readability — a pattern you will see constantly in exam code.

The chain above runs as a single optimised job when .show(10) is called. Spark doesn't execute each line one by one — it plans the whole thing first.

8. Registering a DataFrame as a temporary view

Sometimes you want to write SQL against a DataFrame that isn't a registered Delta table. You can temporarily register it:

df.createOrReplaceTempView("orders_view")

# Now you can query it with SQL
spark.sql("SELECT customer, SUM(amount) FROM orders_view GROUP BY customer")

createOrReplaceTempView only exists for the current Spark session — it is not persisted to disk and disappears when the cluster restarts.

A temp view is session-scoped and not persisted. A Delta table is persisted to disk and survives cluster restarts. If the exam asks which one disappears when the cluster stops — temp view.
