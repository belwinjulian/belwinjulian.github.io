Streaming, Schema Evolution & Lakeflow Connect

1. Structured Streaming — how it actually works

Lesson 3 showed you that Auto Loader runs on top of Spark's Structured Streaming engine. Before you can use it confidently on the exam, you need to understand what that engine is doing.

The mental model: Structured Streaming treats an incoming stream of data as an infinite table. Every time new data arrives, Spark appends rows to that table and runs your query on the new rows only. The result is written to a sink (your Delta table).

Stream of events (Kafka, files, Delta CDC…) │ ▼ ┌──────────────────────┐ │ Micro-batch trigger │ ← wakes up on a schedule or when data arrives │ (process new rows) │ └──────────┬───────────┘ │ ▼ ┌──────────────────────┐ │ Query / Transform │ ← same Spark SQL / DataFrame logic as batch └──────────┬───────────┘ │ ▼ ┌──────────────────────┐ │ Sink (Delta table) │ └──────────────────────┘

The key difference from batch: the job keeps running (or re-runs on a trigger) and only processes new data each time, using the checkpoint to remember where it left off.

2. Output modes

When Structured Streaming writes results to a sink, it needs to know how to write them. There are three output modes:

  • Append — only write new rows to the sink. Nothing already there is changed. This is the default and the most common for Delta tables.
  • Complete — rewrite the entire result table every trigger. Used for aggregations where you want a full summary (e.g. running totals). Expensive on large data.
  • Update — only write rows that changed since the last trigger. More efficient than Complete for aggregations but not all sinks support it.
df.writeStream
  .outputMode("append")          # most common for Delta
  .option("checkpointLocation", "/checkpoints/orders")
  .toTable("orders")
The exam will describe a scenario and ask which output mode to use. Rule of thumb:
— New rows only → append
— Full aggregation result each time → complete
— Only changed aggregation rows → update

3. Trigger modes (recap + new)

Lesson 3 introduced availableNow=True. Here is the full picture of what triggers control:

## Process all available data now, then stop (batch-style run)
.trigger(availableNow=True)

## Wake up every N minutes (continuous micro-batch)
.trigger(processingTime="5 minutes")

## One micro-batch, then stop (older pattern — prefer availableNow)
.trigger(once=True)

## Near-real-time (runs in a single continuous query, low latency)
.trigger(continuous="1 second")
trigger(once=True) and trigger(availableNow=True) look similar but differ: once runs a single micro-batch then stops; availableNow runs all micro-batches until the source is caught up, then stops. availableNow is preferred and more common on the exam.

4. Schema enforcement vs. schema evolution

Delta Lake has strong opinions about schema. Understanding the two modes is a common exam topic.

Schema enforcement (default)

By default, Delta Lake enforces the schema of the table. If you try to write data that doesn't match — extra columns, wrong types — Delta will reject the write and throw an error. This protects your table from accidental corruption.

Schema enforcement is the default behavior in Delta Lake. Writing mismatched data fails unless you explicitly enable evolution.

Schema evolution — mergeSchema

When you want new columns to be accepted automatically, you enable schema evolution. For one-off writes:

df.write \
  .format("delta") \
  .option("mergeSchema", "true") \
  .mode("append") \
  .saveAsTable("orders")

In SQL (COPY INTO or CREATE TABLE AS SELECT):

COPY INTO orders
FROM 's3://my-bucket/raw/'
FILEFORMAT = PARQUET
COPY_OPTIONS ('mergeSchema' = 'true');

Schema evolution — autoMerge (session config)

For streaming or repeated writes where you always want schema evolution on, set it at the Spark session level instead of per-write:

spark.conf.set("spark.databricks.delta.schema.autoMerge.enabled", "true")

With autoMerge enabled, any write that adds new columns will silently extend the Delta table schema instead of failing. Existing columns are never dropped.

Incoming file has columns: id, name, email, phone (new!) ↑ Delta table has: id, name, email mergeSchema = false → ERROR: column "phone" not in schema mergeSchema = true → table gains "phone" column; existing rows get NULL
When a new column appears and the write succeeds — schema evolution is on. Existing rows that predate the new column get NULL for that column. Delta never drops columns automatically.

What schema evolution does NOT do

  • It does not change the type of an existing column (e.g. string → int). That requires an explicit ALTER TABLE.
  • It does not delete columns that are no longer in the source.
  • It does not reorder columns.

5. Schema evolution in Auto Loader

Auto Loader has its own schema evolution story, controlled with cloudFiles.schemaEvolutionMode:

df = (spark.readStream
        .format("cloudFiles")
        .option("cloudFiles.format", "json")
        .option("cloudFiles.schemaLocation", "/checkpoints/schema")
        .option("cloudFiles.schemaEvolutionMode", "addNewColumns")  # default
        .load("s3://my-bucket/raw/"))

The four modes:

  • addNewColumns (default) — new columns are added to the inferred schema; the stream restarts to pick them up
  • rescue — new or mismatched columns are captured in a special _rescued_data JSON column instead of failing
  • failOnNewColumns — throws an error when a new column is detected (strict mode)
  • none — no schema evolution; new columns are silently dropped
The _rescued_data column is an exam favourite. If you see it in answer choices, it means Auto Loader is set to rescue mode — it captured unexpected columns rather than failing or dropping them.

6. Lakeflow Connect

Auto Loader is great for files in cloud storage. But what about data from SaaS systems — Salesforce, Workday, Google Analytics, ServiceNow? Those don't dump files to S3 for you.

Lakeflow Connect (formerly called Databricks Ingest) is Databricks' managed connector service. It provides pre-built, no-code connectors that pull data from SaaS sources directly into your Lakehouse.

SaaS Sources ┌────────────┐ ┌─────────────┐ ┌──────────────┐ │ Salesforce │ │ Google Ads │ │ ServiceNow │ … └─────┬──────┘ └──────┬──────┘ └──────┬───────┘ │ │ │ └────────────┬────┘──────────────────┘ │ Lakeflow Connect (managed connectors) │ ▼ Delta Table (Bronze) in Unity Catalog

What Lakeflow Connect handles for you

  • Authentication to the SaaS API
  • Incremental sync — only pulls changed records, not the full dataset every time
  • Schema mapping — SaaS fields → Delta columns
  • Error handling and retries
  • Scheduling — runs on a configurable cadence via Lakeflow Jobs

When to use Lakeflow Connect vs. Auto Loader

LAKEFLOW CONNECT AUTO LOADER ────────────────────────────────────────────────────────── Source SaaS APIs Files in cloud storage Setup No-code / UI Python code Managed by Databricks You (checkpoints, triggers) Schema Auto-mapped from API Inferred from files Use case Salesforce, Workday… CSV / JSON / Parquet drops
On the exam: if the scenario mentions a SaaS source (Salesforce, Google Analytics, SAP, etc.) and asks how to ingest it — the answer is Lakeflow Connect, not Auto Loader. Auto Loader is for files in cloud storage.
