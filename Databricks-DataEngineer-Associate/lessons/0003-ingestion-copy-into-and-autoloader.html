Ingestion: COPY INTO & Auto Loader

1. The ingestion problem

Data engineering almost always starts with the same question: how do I get files from cloud storage into a Delta table? Files arrive as CSV, JSON, Parquet, etc. in S3, Azure Data Lake, or GCS. You need to load them into Bronze.

Databricks gives you two tools for this. Choosing between them is a common exam question.

Cloud Storage (S3 / ADLS / GCS) ┌──────────────────────────────────┐ │ orders_2026-06-01.csv │ │ orders_2026-06-02.csv │ │ orders_2026-06-03.csv ← new │ └──────────┬───────────────────────┘ │ ┌──────┴──────┐ │ │ COPY INTO Auto Loader (batch SQL) (streaming Python) │ │ └──────┬──────┘ ▼ Delta Table (Bronze)

2. COPY INTO — batch SQL ingestion

COPY INTO is a SQL command that loads files from a cloud storage path into a Delta table. Its killer feature: it is idempotent. Run it ten times on the same folder and it only loads each file once — it tracks which files it has already processed and skips them.

COPY INTO orders
FROM 's3://my-bucket/raw/orders/'
FILEFORMAT = CSV
FORMAT_OPTIONS ('header' = 'true', 'inferSchema' = 'true')
COPY_OPTIONS ('mergeSchema' = 'true');

Breaking that down:

  • COPY INTO orders — target Delta table
  • FROM 's3://...' — source path (S3, ADLS, or GCS)
  • FILEFORMAT = CSV — supported formats: CSV, JSON, PARQUET, AVRO, ORC, TEXT, BINARYFILE
  • FORMAT_OPTIONS — file-specific settings (e.g. header row, delimiter)
  • COPY_OPTIONS ('mergeSchema' = 'true') — allow new columns (schema evolution)
COPY INTO is idempotent — re-running it on the same path is safe. It skips already-loaded files. This is the word the exam uses: idempotent.

When to use COPY INTO

  • Loading a bounded, known set of files (e.g. a one-time historical load)
  • When you want SQL-only — no Python needed
  • Hundreds to low thousands of files — manageable scale

3. Auto Loader — incremental streaming ingestion

Auto Loader is Databricks' answer to continuous ingestion: files land in cloud storage and are automatically picked up and processed as a stream. You do not poll for new files — Auto Loader watches the folder for you.

It uses Spark's Structured Streaming under the hood, written in Python:

df = (spark.readStream
        .format("cloudFiles")              # Auto Loader source
        .option("cloudFiles.format", "csv")
        .option("cloudFiles.schemaLocation", "/checkpoints/orders_schema")
        .load("s3://my-bucket/raw/orders/"))

df.writeStream
  .option("checkpointLocation", "/checkpoints/orders")
  .trigger(availableNow=True)             # or .trigger(processingTime="10 minutes")
  .toTable("orders")

Key things to notice:

  • .format("cloudFiles") — this is the Auto Loader source format. cloudFiles is the magic keyword.
  • cloudFiles.format — the format of the incoming files (csv, json, parquet…)
  • cloudFiles.schemaLocation — where Auto Loader saves the inferred schema so it doesn't have to re-infer every run
  • checkpointLocation — where Auto Loader tracks which files it has already processed
If you see format("cloudFiles") in exam code — that is Auto Loader. That string is the identifier.

The checkpoint — how Auto Loader remembers what it processed

The checkpointLocation is critical. Auto Loader writes a checkpoint file every time it processes a batch of new files. If the job restarts, it reads the checkpoint and continues from exactly where it left off — no duplicate processing, no missed files.

If a question says "Auto Loader restarted and reprocessed all files from the beginning" — the checkpoint location was missing or deleted.

Schema inference and evolution in Auto Loader

Auto Loader can infer the schema of incoming files automatically and store it at cloudFiles.schemaLocation. When a new column appears in a future file, it can evolve the schema automatically — no manual intervention needed.

When to use Auto Loader

  • Files arrive continuously over time (daily drops, real-time feeds)
  • Millions of files — Auto Loader scales far beyond COPY INTO
  • You need automatic schema inference and evolution
  • You want the pipeline to run continuously or on a trigger

4. COPY INTO vs. Auto Loader — the exam comparison

COPY INTO AUTO LOADER ───────────────────────────────────────────────────── Language SQL Python (Streaming) Best for Bounded file sets Continuous / millions of files Idempotent ✓ Yes ✓ Yes (via checkpoint) Schema inference Manual Automatic (cloudFiles.schemaLocation) Scale Hundreds of files Millions of files Trigger Manual / scheduled Continuous or triggered Keyword COPY INTO format("cloudFiles")
The exam will give you a scenario and ask which to use. The decision rule:
— Known, finite set of files + SQL preference → COPY INTO
— Files arriving continuously + large scale → Auto Loader

5. Trigger modes in Auto Loader

Auto Loader does not have to run 24/7. You control how often it wakes up with .trigger():

## Process all available files right now, then stop
.trigger(availableNow=True)

## Run every 10 minutes
.trigger(processingTime="10 minutes")

## Run continuously (default if no trigger set)
.trigger(continuous="1 second")
availableNow=True is the most exam-tested trigger. It means: process everything currently in the folder, then stop. Useful for scheduled batch-style runs using a streaming API.
